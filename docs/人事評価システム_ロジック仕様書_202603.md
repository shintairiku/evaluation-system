### 3-2. 各社員行の数式（行8〜100、パターンは行番号Nで共通）

#### 入力値（手動入力）
```
G[N] : 社員番号（文字列、例: "00001"）
H[N] : 氏名
I[N] : 部署
J[N] : MBO最終評価 ← **BR153（上長評価）を使用（確定）**（"SS","S","A+","A","A-","B","C","D" のいずれか）
K[N] : MBOウェイト（固定値: 100）
M[N] : コンピテンシー最終評価（同上）
N[N] : コンピテンシーウェイト（固定値: 10）
P[N] : コアバリュー最終評価（同上）
V[N] : 現在ステージ（例: "STAGE5"、手動入力）
```

#### 計算式（全行共通パターン）

```excel
L[N] = VLOOKUP(J[N], BD:BE, 2, 0) * (10/11)
```
> MBO最終評価を点数（0〜7）に変換し、10/11を乗算。
>
> **10/11の根拠:**
> MBOのウェイト = 100%、コンピテンシーのウェイト = 10%、合計 = **110%**
> 正規化すると MBO = 100/110 = **10/11**、コンピテンシー = 10/110 = **1/11**
> つまり最終的にMBOが全体の約90.9%、コンピテンシーが約9.1%を占める設計。

```excel
O[N] = VLOOKUP(M[N], BD:BE, 2, 0) * (1/11)
```
> コンピテンシー最終評価を点数（0〜7）に変換し、1/11を乗算。
> ウェイト比率：コンピテンシー = 10/110 = 1/11 ≈ 9.091%（10/11の根拠は上記参照）

```excel
Q[N] = L[N] + O[N]
```
> 合計点（理論最大値: 7 × (10/11 + 1/11) = 7.0）

```excel
R[N] = VLOOKUP(Q[N], BG:BH, 2, 1)
```
> 合計点→総合評価への変換（近似一致）。セクション2-2の変換表を使用。

```excel
S[N] = IF(J[N]="D", "D", 0)
```
> MBO最終評価が"D"の場合 → `"D"`、それ以外 → `0`
> **確定済み:** フラグ値は `"D"` で統一。Excelの行8にある `"D評価"` は入力ミスのため無視する。

```excel
T[N] = IF(
  AND(
    OR(R[N]="A+", R[N]="S", R[N]="SS"),
    OR(M[N]="A+", M[N]="S", M[N]="SS"),
    OR(P[N]="A+", P[N]="S", P[N]="SS")
  ),
  "昇格",
  ""
)
```
> 昇格条件：総合評価(R)・コンピテンシー(M)・コアバリュー(P) の3つ全てが A+, S, SS のいずれか → "昇格"
> 1つでも A以下があれば空文字

```excel
U[N] = VLOOKUP(R[N], レベル増減マスタ!B:C, 2, 0)
```
> 総合評価からレベル増減値を取得（完全一致）。セクション2-3の表を使用。

```excel
W[N] = VLOOKUP(G[N], レベル貼付シート!C:D, 2, FALSE)
```
> 社員番号から現在レベルを取得（完全一致）。

```excel
X[N] = U[N] + W[N]
```
> 新レベル = レベル増減値 + 現在レベル
> **アラート条件:** X[N] ≥ 31 の場合は警告表示（Excelの条件付き書式で実装）

### 4-4. 目標ごとの計算数式

各目標（i=1〜7）の代表行を [ROW] とする（目標1=行40、目標2=行55、…目標7=行130）:

```excel
BX[ROW] = VLOOKUP(BI[ROW], CF:CG, 2, 0)
```
> 自己評価（BI列）を点数（BX列）に変換（完全一致）

```excel
BY[ROW] = (AU[ROW] / 5) * BX[ROW]
```
> 自己評価の加重点数 = (目標ウェイト% ÷ 5) × 自己評価スコア
>
> **計算の意味:** スコアの最大値は5（SS）。ウェイト÷5は「この目標がSS達成時に何点を生む」を意味する。
> 例: 定量目標（ウェイト70%）でSS評価（5点）→ 70÷5×5 = **70点**
> 例: 定性目標（ウェイト30%）でS評価（4点）→ 30÷5×4 = **24点**
> 全目標がSSの場合、合計は **100点**（満点）。

```excel
BZ[ROW] = VLOOKUP(BV[ROW], CF:CG, 2, 0)
```
> 上長評価（BV列）を点数（BZ列）に変換（完全一致）

```excel
CA[ROW] = (AU[ROW] / 5) * BZ[ROW]
```
> 上長評価の加重点数 = (目標ウェイト% ÷ 5) × 上長評価スコア（BY列と同じロジック）

### 5-7. 全体集計・最終評価数式

> ⚠️ **重要: Excelの実装とシステム推奨実装が異なる**

#### Excelの実装（平均フラット方式）
```excel
CL77 = AVERAGE(CL35:CL64)
```
全30項目の本人評価数値を単純平均。項目数が多い大項目の影響が大きくなる。

```excel
CM77 = AVERAGE(CM35:CM64)
```
全30項目の上長評価数値を単純平均。

#### ✅ システム推奨実装（2段階平均方式）
**PDFの仕様が正しい。** 大項目ごとの項目数が異なる場合（特にパートタイマー）、単純平均では大項目間の公平性が損なわれる。

```
Step 1: 大項目ごとに小項目スコアの平均を算出 → 大項目スコア（1〜6）
Step 2: 6つの大項目スコアの平均を算出 → コンピテンシー最終スコア
```

擬似コード:
```
group_scores = []
for each group (1〜6):
    group_score = AVERAGE(items in group)
    group_scores.append(group_score)

final_score = AVERAGE(group_scores)
final_rating = VLOOKUP(final_score, threshold_table, approximate)
```

**根拠:** 全項目数が同じ場合（各グループ5項目）は結果が一致するが、パートタイマーや将来的な項目数変更に対して堅牢な設計になる。

```excel
AB80 = IFERROR(VLOOKUP(CL77, CR:CS, 2, 1), "")
```
> 本人総平均→最終コンピテンシー評価（本人）（近似一致）※参考値、総合評価には不使用

```excel
CD80 = IFERROR(VLOOKUP(CM77, CR:CS, 2, 1), "")
```
> 上長総平均→最終コンピテンシー評価（上長）（近似一致）← **★総合のM列に使用するのはこちら**

## 8. ビジネスルールまとめ

### ルール1: MBOウェイト構成（ステージ別）
- 定量目標のウェイトはステージによって異なる（70% / 80% / 100%）
- 定性目標のウェイト = 100% − 定量ウェイト（0% / 20% / 30%）
- 合計は常に100%
- 詳細はセクション4-6の表を参照

### ルール2: ★総合での評価ウェイト（固定、変更不可）
- MBO: 10/11（≈90.909%）
- コンピテンシー: 1/11（≈9.091%）
- コアバリュー: 点数計算には不算入（昇格判定のみに使用）
- **ウェイトの導出:** MBO=100% + コンピテンシー=10% → 合計110%。正規化するとMBO=100/110=10/11、コンピテンシー=10/110=1/11

### ルール3: 昇格条件（全て満たす必要あり）
- 総合評価（R列）が A+, S, SS のいずれか
- コンピテンシー評価（M列）が A+, S, SS のいずれか
- コアバリュー評価（P列）が A+, S, SS のいずれか

### ルール4: D評価強制ルール（✅ 確定・システム対応済み）
- **確定:** 半年ごとの評価において、MBOが「D」の場合は降格となる
- 上期でD評価 → 降格、下期でD評価 → 降格（各半期は独立して評価）
- cross-period（上期＋下期の組み合わせ）チェックは不要
- **システム対応状況:** MBO=D → 総合スコアが低下 → overall_rank=D → 降格フラグ発動。自動的に正しく動作する

### ルール5: 新レベルアラート（確定済み）
- 新レベル（X列）≥ 31 の場合はアラートを表示
- 上限値（キャップ）は存在しない — レベルは上限なしで計算される
- アラートはHRへの通知のみ。システムによる自動処理は不要
- アラート受信後はHRが手動でステージ昇格・職種変更等を検討する
- **根拠:** X列の数式 `=U[N]+W[N]` にキャップ処理なし。X5ヘッダーも「アラート」のみ

### ルール6: コアバリュー評価者の除外ルール
- 最終コアバリュー評価（T27）は同僚1・同僚2・上司の平均
- 本人評価（Q列・U列）は最終評価には使用しない

### ルール8: D=0による境界値処理（システム必須実装）

D=0の採用により、変換テーブルの最小閾値を下回るケースが発生する。以下の2つのルールを必ずシステムに実装すること。全64通りのMBO×コンピテンシー組み合わせで検証済み。

**ルールA — ★総合 合計点（Q列）の下限処理**
```
if Q < 0.1:
    total_rating = "D"
```
対象ケース: MBO=D かつ コンピテンシー=D（Q=0）、MBO=D かつ コンピテンシー=C（Q=0.09）

**ルールB — MBOシート 合計点の下限処理**
```
if mbo_total < 20:
    mbo_rating = "D"
```
対象ケース: 全目標がD評価で合計点が0になる場合

**適用順序（優先度順）:**
1. ルールB（MBOシート内）: 合計点 < 20 → MBO評価 = "D"
2. ルールA（★総合内）: Q < 0.1 → 総合評価 = "D"
3. ルール4（Dオーバーライド）: 両方の半期MBO=D → 総合評価を"D"に強制（最優先）

### ルール7: パートタイマー・一般スタッフの扱い（確定済み）

MBOウェイトはセクション4-6の表の通り、全て定量100%・定性0%。
コンピテンシーのウェイトは社員と同じ10%（1/11）。
評価段階はMBO・コンピテンシー・コアバリューともに社員と同じ段階を使用。

| 区分 | 定量 | 定性 | コンピテンシー |
|---|---|---|---|
| パートリーダー上級/中級/初級 | 100% | 0% | 10% |
| 一般スタッフ上級/中級/初級 | 100% | 0% | 10% |

## 14. ✅ 解決済みケース一覧（元・未定義ケース）

### ✅ 解決済みケース3: 新レベルの上限（キャップ）

**状況:** アラートはレベル≥31で発動するが、最大レベルが定義されていないと思われていた。

**解決:** Excelの数式 `X[N] = U[N] + W[N]` にはIF・MIN等の上限処理が一切なく、X5のヘッダーも「アラート」のみ。上限値は存在しない。

**確定ルール:**
- 新レベルは上限なしで計算する（`新レベル = 現在レベル + 増減値`）
- 新レベル ≥ 31 の場合はアラートを表示し、HRが手動対応する（ステージ昇格・職種変更等の検討）
- システムはアラートフラグを立てるのみ。自動キャップや自動処理は不要。

---

### ✅ 解決済みケース4: 列J（MBO最終評価）のデータソース — Y153 vs BR153

**状況:** ★総合シートのJ列（MBO最終評価）が、MBOシートのY153（自己評価）とBR153（上長評価）のどちらを参照すべきか未定義だった。

**解決:** **BR153（上長評価）を使用する。**

**根拠（3点）:**
1. **コンピテンシーとの一貫性:** コンピテンシー評価（M列）は仕様書セクション5-7で明確に「CD80(上長) ← ★総合のM列に使用するのはこちら」と定義されている。MBOも同様に上長評価を使用するのが論理的に一貫している。
2. **評価フローの原則:** 人事評価制度において、自己評価は参考値であり、上長が最終的な評価を決定・承認する。最終評価に自己評価を使用することは制度の趣旨に反する。
3. **既存システム実装との整合:** 現行システムの総合評価SQLクエリ（comprehensive_evaluation_repo.py）は既にsupervisor_rating（上長評価）のみを使用しており、self_ratingは参照していない。

**確定ルール:**
- J[N]（MBO最終評価）= BR153（上長評価によるMBO最終評価）
- Y153（自己評価によるMBO最終評価）は参考値としてのみ表示し、総合評価の算出には使用しない

| ケース | 処理 | 根拠 |
|---|---|---|
| MBO未提出（mbo_total_100 = NULL） | performance_rank = NULL, total_score = NULL, overall_rank = NULL | 片方のカテゴリが未確定では総合評価を算出できない |
| コンピテンシー未提出（competency_raw_score = NULL） | competency_rank = NULL, total_score = NULL, overall_rank = NULL | 同上 |
| コアバリュー評価者不足（同僚2名+上司の3名未満） | core_value_rank = NULL | 仕様の3名平均（V27）が計算不能 |
| コアバリューのピアが1名のみ | core_value_rank = NULL（2名+上司=3名が必須） | 仕様に忠実 |
| コアバリューのピアが0名 | core_value_rank = NULL | 仕様に忠実 |
| 目標ウェイトが100%未満（下書き中） | mbo_total_100は入力済み分のみで計算される | 提出時に100%バリデーション済み |

### 15-2. 丸め処理（Rounding rules）

| 対象 | ルール | 実装 |
|---|---|---|
| total_score（Q列相当） | 小数点以下2桁に丸め | Python: `round(q, 2)`（banker's rounding） |
| mbo_total_100 | 小数点以下2桁に丸め | SQL: `ROUND(mbo_total_100, 2)` |
| competency_raw_score | 小数点以下2桁に丸め | SQL: `ROUND(competency_raw_score, 2)` |
| core_value_raw_score | 小数点以下2桁に丸め | SQL: `ROUND(core_value_raw_score, 2)` |

**注意:** banker's roundingにより、0.5の境界値は偶数側に丸められる（例: 2.65 → 2.6, 2.75 → 2.8）。閾値判定は丸め後の値で行う。

### 15-3. 新レベルの下限処理

```
新レベル = 現在レベル + 増減値
処理時: max(1, min(30, 新レベル))
```

- 下限: 1（レベル0以下にはならない）
- 上限: 30（処理時にクランプ。≥31はアラートのみで自動処理はしない）
- **根拠:** `comprehensive_evaluation_service.py` の `USER_LEVEL_MIN=1, USER_LEVEL_MAX=30`

## 16. 解決済みルール（ビジネスオーナー確認済み 2026年4月3日）

### 16-1. ルール4: D評価 → 降格（✅ 解決済み・システム対応済み）

**ビジネスオーナー確認（2026年4月3日）:**
> 現在は1年ではなく、半年ごとに評価をしています。上期でD評価になれば降格になるし、下期でDになっても降格です。

**結論:** 各半期は独立して評価される。cross-period（上期＋下期の組み合わせ）チェックは不要。
**システム対応:** MBO=D → mbo_total_100が低下 → performance_rank=D → overall_rank=D → 降格フラグ発動。**自動的に正しく動作する。追加実装不要。**

### 16-2. コンピテンシー2段階平均方式（✅ 解決済み・実装済み）

**ビジネスオーナー確認（2026年4月3日）:**
> 各グループの小項目の平均を算出します。その後にグループごとの平均を算出します。その結果がコンピテンシーの総合評価となります。

**実装:**
- Stage 1（小項目 → 大項目平均）: `supervisor_feedback_repo.py` の `_calculate_supervisor_rating_from_rating_data()` が各Goal（=大項目）のrating_data内の全アクション（=小項目）を平均化してsupervisor_ratingに保存
- Stage 2（大項目平均 → 最終スコア）: `comprehensive_evaluation_repo.py` の `AVG(gf.supervisor_rating)` が全コンピテンシーGoalの単純平均を算出
- **加重平均ではなく単純平均**を使用（各大項目が均等に評価される）
