# ステージ・レベル・給与（基本給/時給）仕様（総合評価連携）

## 1. 目的 / 背景

総合評価（業績目標 + コンピテンシー）の結果から **レベル増減（正社員）** を決定し、その結果を **基本給（正社員）** に反映できるようにする。

また、ステージ（グレード）は正社員・パートタイム共通の等級として保持し、ステージ変更（昇格/降格等）が発生した場合に **基本給（正社員）** / **時給（パートタイム）** の参照先となる。

本仕様は「総合評価テーブル」で表示される **レベル増減** を、給与改定に接続するためのルールを定義する。

---

## 2. 用語定義

| 用語 | 意味 |
|---|---|
| 雇用形態 | 正社員（Employee） / パートタイム（Part-time） |
| ステージ（グレード） | 両雇用形態に存在する等級。昇格/降格の単位。 |
| レベル | **正社員のみ**が持つ、ステージ内の序列（1〜30）。 |
| 基本給 | 正社員の月額（または年俸換算前の基準額）を指す前提の金額。単位は円。 |
| 時給 | パートタイムの時間あたり単価。単位は円。 |

---

## 3. 要件概要（必須）

### 3.1 ステージ（グレード）

- ステージは **正社員・パートタイム共通**で存在する
- 本機能（総合評価の算出）により **ステージが自動で増減するものではない**
  - ステージ変更（昇格/降格など）は別プロセス（人事確定/運用）で行い、確定したステージを本仕様の給与算出が参照する

### 3.2 レベル（正社員のみ）

- 正社員は、各ステージに **レベル1〜30** を持つ
- 総合評価（業績目標 + コンピテンシー）により、**レベル増減（例: `+8`, `-5`）** が決まる
  - 総合評価シートの「レベル増減」列に相当し、システムでも算出して表示する

### 3.3 パートタイム（レベルなし）

- パートタイムにはレベルを持たせない
- パートタイムは、ステージが上がると時給が上がる（ステージが下がると時給が下がる）設計とする

---

## 4. 正社員：基本給の算出仕様（必須）

### 4.1 ステージ別「レベルあたり増額」

正社員は「ステージごとに、レベルが1上がるたびに増える基本給」が異なる。

例:
- ステージ1: レベルが1上がるごとに基本給 +2,000円
- ステージ2: レベルが1上がるごとに基本給 +3,000円
- ステージが上がるほど、増額幅も大きくなる

### 4.2 必要なマスタ項目（提案）

正社員向けに、ステージごとに以下を保持する。

| 項目 | 例 | 備考 |
|---|---:|---|
| `level_min` | 1 | 固定でよい（将来拡張で可変も可） |
| `level_max` | 30 | 固定でよい（将来拡張で可変も可） |
| `base_salary_level1_yen` | 240,000 | 当該ステージのレベル1の基本給 |
| `per_level_increment_yen` | 2,000 | レベルが1増えるごとの増額（ステージ別） |

### 4.3 基本給の計算式（提案）

`stage` と `level` が確定している場合、基本給を以下で算出する。

- `base_salary_yen = base_salary_level1_yen(stage) + (level - 1) * per_level_increment_yen(stage)`

### 4.4 レベル範囲外の扱い（必須）

総合評価により算出された `新レベル` が範囲外の場合（例: `>= 31`、`<= 0`）は、以下の扱いとする。

- 管理者テーブル上は **アラート（強調表示）** を行う
- 給与（新基本給）の自動算出は **未確定（`null`）扱い** とし、手動確認が必要な状態にする

> 既存仕様の「新レベル（31以上はアラート）」と整合させる。

### 4.5 総合評価ランク別のレベル増減（必須）

総合評価（SS〜D）に応じたレベル増減は、以下をデフォルト（基準）として扱う。

| 総合評価 | レベル増減 |
|---|---:|
| SS | +10 |
| S | +8 |
| A+ | +6 |
| A | +5 |
| A- | +2 |
| B | +1 |
| C | -5 |
| D | -8 |

※ 実装では `evaluation_score_mapping`（「＜総合評価・点数対応表＞」）に保持し、`eval_admin` が変更可能な形にする想定。

---

## 5. パートタイム：時給の算出仕様（必須）

### 5.1 必要なマスタ項目（提案）

パートタイム向けに、ステージごとに以下を保持する。

| 項目 | 例 | 備考 |
|---|---:|---|
| `hourly_wage_yen` | 1,300 | ステージに紐づく時給 |

### 5.2 時給の決定ルール（必須）

- `時給 = hourly_wage_yen(ステージ)`
- パートタイムはレベルを持たないため、総合評価テーブル上の **レベル増減は表示しない**（もしくは `-` 表示）

---

## 6. 総合評価（Comprehensive Evaluation）との連携

### 6.1 入力（必要な前提データ）

総合評価テーブル上で給与改定に接続するために、行データとして以下が必要となる。

| 項目 | 正社員 | パートタイム |
|---|---:|---:|
| 現在ステージ | ✅ | ✅ |
| 現在レベル | ✅ | ❌ |
| レベル増減（算出） | ✅ | ❌ |
| 新レベル（算出） | ✅ | ❌ |

### 6.2 算出フロー（提案）

1. 総合評価（業績 + コンピテンシー）から `evaluation_score_mapping` を参照し、正社員の `レベル増減` を決定する
2. 正社員のみ `新レベル = 現在レベル + レベル増減` を算出する
3. 正社員は `（現在ステージ, 新レベル）` から新基本給を算出する（範囲外なら未確定）
4. パートタイムは `現在ステージ` から時給を決定する（ステージ変更が別途確定した場合は、その確定後ステージを参照する）

> ステージ変更（昇格/降格等）が別途発生する場合、その確定後ステージをもとに同様の式で給与を算出する。

### 6.3 表示要件（提案）

総合評価テーブルに以下の列を追加できる（給与改定を画面で確認する用途）。

- 正社員: `現在基本給`, `新基本給`, `差分（円）`
- パートタイム: `現在時給`, `新時給`, `差分（円）`

※ 本列追加はUI要件として別途確定する（当面はバックエンドDTO/計算ロジックの設計のみでも可）。

---

## 7. データモデル（提案）

既存の `stages` が評価ウェイト等を持っているため、給与マスタは以下いずれかの方式で保持する。

### 7.1 方式A：`stages`に給与項目を追加（シンプル）

- 長所: テーブルが増えない
- 短所: 正社員/パートで必要項目が異なり `NULL` が増える、関心の混在

### 7.2 方式B：給与マスタを分離（推奨）

例: `stage_compensation_rules`

| カラム（例） | 型（例） | 説明 |
|---|---|---|
| `stage_id` | FK | ステージ |
| `employment_type` | enum | `employee` / `part_time` |
| `level_min` | int | 正社員のみ |
| `level_max` | int | 正社員のみ |
| `base_salary_level1_yen` | int | 正社員のみ |
| `per_level_increment_yen` | int | 正社員のみ |
| `hourly_wage_yen` | int | パートタイムのみ |

---

## 8. 未決定事項 / 要確認（要件確定が必要）

- ステージの一覧（正社員/パートタイムで共通か、別マスタか）
- 各ステージの `base_salary_level1_yen` と `per_level_increment_yen` の具体値（ステージ1=2,000円/レベル、ステージ2=3,000円/レベル…のような増額幅の一覧）
- ステージが変わる場合のレベルの扱い
  - 例: レベルを維持する / レベルをリセットする / ルール（上限/下限）を設ける
- レベル範囲外（31以上等）の給与確定フロー（自動キャップするか、必ず手動確定か）
