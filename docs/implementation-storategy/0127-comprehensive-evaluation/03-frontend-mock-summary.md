## 6. 算出・判定ロジック（モック実装）

### 6.1 合計点

- `合計点 = 業績点 + コンピテンシー点`
- どちらかが未確定（`null`）の場合は、合計点は未表示（`-`）

### 6.2 総合評価（合計点 → ランク）

合計点から、以下の「＜総合評価・点数対応表＞」で **総合評価（SS〜D）** を決定します。

| 総合評価 | 点数レンジ |
|---|---|
| SS | 6.5点以上 |
| S | 5.5点以上 〜 6.5点未満 |
| A+ | 4.5点以上 〜 5.5点未満 |
| A | 3.7点以上 〜 4.5点未満 |
| A- | 2.7点以上 〜 3.7点未満 |
| B | 1.7点以上 〜 2.7点未満 |
| C | 1.0点以上 〜 1.7点未満 |
| D | 0.1点以上 〜 1.0点未満 |

### 6.3 MBO D評価フラグ（総合評価の上書き）

- 設定が有効な場合、`MBO D評価フラグ = 1` のとき、総合評価は **Dに上書き** します
  - 前提: `MBO D評価フラグ`は「同一年度の上期と下期のMBOが両方D」のときに `1` となる年次判定用フラグ
  - 背景ルール: 「上期と下期のMBOが両方Dの場合、コンピテンシー/クレドにかかわらず（年次の）総合評価はD」
- 注: 上の総合評価テーブルは半期ごとの評価を表す想定のため、年次集約が未実装のマイルストーン1ではフラグはモック値（表示のみ）として扱います

### 6.4 昇格フラグ（新レベルが30以上）

- 正社員のみ、`新レベル >= 30` の場合は **昇格フラグを点灯**します（常に）
  - パートタイムはレベルの概念がないため対象外
- 昇格フラグは「ステージ変更（アップ/ダウン）を検討・確定する必要がある」ことを示す **起点フラグ**です
  - ステージ（グレード）の自動更新は行いません
- 昇格フラグ点灯者に対するステージ変更（アップ/ダウン）と反映後レベル（正社員のみ）は、`eval_admin`が `/admin-eval-list/candidates` の「編集（確定）」から **手動で確定**します（理由・ダブルチェック者必須）

### 6.4.2 降格フラグ（総合評価がD）

- `総合評価 = D` の場合、**降格フラグを点灯**します（評価期間ごとに即時）
  - 「年間集約」は前提としない（単一評価期間でDなら点灯）
- 降格フラグも「ステージ変更（アップ/ダウン）を検討・確定する必要がある」ことを示す **起点フラグ**です
  - ステージ（グレード）の自動更新は行いません

### 6.5 ステージ変更（アップ/ダウン）の手動確定

- 昇格フラグ点灯者に対して、`eval_admin` が判定（昇格/降格/対象外）を手動で選択し、反映後ステージ・反映後レベル（正社員のみ）を入力して確定します
- 反映後レベルは **1〜30の範囲**で確定する想定です（範囲外のまま確定させない）

### 6.6 レベル増減と新レベル

- レベル増減
  - **総合評価（SS〜D）に応じたデフォルト増減** を適用（設定で変更可能）
  - パートタイムにはレベルの概念がないため、レベル増減は `-` 表示とする

| 総合評価 | レベル増減（デフォルト） |
|---|---:|
| SS | +10 |
| S | +8 |
| A+ | +6 |
| A | +5 |
| A- | +2 |
| B | +1 |
| C | -5 |
| D | -8 |

- 新レベル
  - `新レベル = 現在レベル + レベル増減`
  - `新レベル >= 31` の場合、アラート表示（強調）
  - パートタイムは `現在レベル/新レベル` ともに `-` 表示（アラート対象外）
  - 注: ステージ変更（昇格/降格）を確定する場合の「反映後レベル」は、上記の算出値とは独立に **手動入力で確定** します

## 7. eval_admin向け「判定ルール設定」（/admin-eval-list）

### 7.1 変更できる項目（モック）

- MBO D評価フラグで総合評価をDに上書きするか（ON/OFF）
- レベル増減（総合評価別）
  - SS/S/A+/A/A-/B/C/D それぞれの増減値
- 補足: 昇格フラグは「新レベルが30以上」で常に点灯、降格フラグは「総合評価がD」で常に点灯とし、設定で変更しない

### 7.2 保存方法（重要）

- 【マイルストーン1（モック）】設定は **ブラウザ（localStorage）に保存** されます（デモ/レビュー用）
  - サーバーやDBには保存されません（他ユーザー/他端末へ共有されません）
- 【本番（Production想定）】`eval_admin`による設定変更は **サーバー/DBに永続化** し、組織内で共有される必要があります
  - 想定DB:
    - `evaluation_score_mapping`（点数→ランク、総合評価別`level_delta`等）
    - 複合条件ルール（`AND`/`OR`）用の別テーブル（例: `comprehensive_evaluation_rules`）

## 8. 昇格/降格フラグ対応ページ（/admin-eval-list/candidates）

### 8.1 目的

昇格フラグ（新レベル>=30）または降格フラグ（総合評価=D）が点灯した対象を一覧表示し、`eval_admin`がステージ変更（アップ/ダウン）と反映後レベル（正社員のみ）を手動で確定できるようにします。

### 8.2 機能（モック）

- フィルタ/検索（評価期間・部署・ステージ・雇用形態（正社員・パート）・表示種別・検索）
- 「フラグのみ」トグル（フラグ未点灯を非表示）
- フラグ点灯件数 / 昇格フラグ件数 / 降格フラグ件数 / 昇格確定件数 / 降格確定件数の表示
- 一覧上で雇用形態がぱっと見て判別できるように表示（例: 正社員/パートのバッジ）
  - パートタイムにはレベルの概念がないため、レベル関連（現在レベル/レベル増減/新レベル）は `-` 表示（アラート対象外）
- `eval_admin`のみ:
  - リーダー面談 / 事業部長プレゼン / CEO面談のチェック更新（マイルストーン1ではブラウザ保存。本番ではDB保存）
  - 特例反映（手動確定）
    - 判定（昇格/降格/対象外）を入力して確定
    - 昇格/降格を確定する場合は **反映後ステージ** を手動で指定して確定
    - 正社員の場合は **反映後レベル（1〜30）** も手動で指定して確定（増減入力ではなく確定値入力）
    - 理由とダブルチェック者が必須（モックではブラウザ保存。本番では監査ログ含めDB保存）

## 9. 次マイルストーン（承認後）の実装ポイント（提案）

- `evaluation_score_mapping`の設計確定（「＜最終評価・点数対応表＞」と「＜総合評価・点数対応表＞」の保持方法）
  - 「総合評価（SS〜D）別のレベル増減（SS:+10 / S:+8 / A+:+6 / A:+5 / A-:+2 / B:+1 / C:-5 / D:-8）」は、`evaluation_score_mapping`の「＜総合評価・点数対応表＞」に`level_delta`として保持する想定
- 年間判定（上期+下期の集約）をバックエンド集計で一貫実装
- クリア条件（面談/プレゼン/CEO面談）の永続化（DB保存）と監査ログ
- 特例反映（手動上書き）の永続化（DB保存）と監査ログ（誰が・いつ・なぜ・ダブルチェック者）
- 権限（`admin`閲覧のみ / `eval_admin`編集可）の正式運用（API側で強制）
- 判定ルール（昇格/降格/上書き）を `AND`/`OR` を含む条件式として永続化し、Backendで評価する
  - `evaluation_score_mapping`（点数→ランク/デフォルト増減）とは責務を分離し、ルール定義は別テーブル（JSONB等）で持つ想定
