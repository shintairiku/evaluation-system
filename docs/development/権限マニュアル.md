# 権限マニュアル — Permissions Master Manual

このマニュアルは、人事評価システムの **権限システム（Permissions System）** の完全リファレンスです。`/org-management` の「権限」タブにある権限マトリクスを安全に運用・編集するために、各 permission の意味、影響範囲、関連コードを記載しています。

> **このドキュメントが存在する理由**: 元の権限システム実装者が退職し、設計意図が文書化されていなかったため、チームが「権限タブを触ると何が壊れるか分からない」と感じていました。本マニュアルは、その不安を解消するために作成されました。

## 全体像 — 3層の認可アーキテクチャ

リクエストは3つの認可レイヤーを通過します。各レイヤーは独立しており、フェイルセーフ（Defense in Depth）として機能します。

```
リクエスト
  ↓
[1] Clerk Middleware（frontend/src/middleware.ts）
    └─ org:admin チェック（フロントエンドの管理者ルート保護）
  ↓
[2] Server Guard（page.tsx の React Server Component）
    └─ getCurrentUserContextAction でロールを取得し redirect 判定
  ↓
[3] Backend AuthContext（backend/app/security/dependencies.py）
    └─ DB の role_permissions テーブルから動的に権限を解決
    └─ @require_permission / @require_role / context.has_permission() で判定
```

### 何が動的か、何が静的か

- **動的（マトリクスが制御）**: バックエンドが `Permission.USER_MANAGE` などの permission code をチェックする箇所はすべて、DB の `role_permissions` テーブルを参照します。UI で chec/uncheck すると即座に反映されます。
- **静的（マトリクスが制御しない）**: フロントエンドのミドルウェアで `orgRole === 'org:admin'` をチェックする箇所、およびバックエンドで `has_role("admin")` のようにロール名を直接チェックする箇所は、マトリクスを参照しません。**第5章「ハードコード例外」** に網羅されています。

## ロール階層

[frontend/src/utils/hierarchy.ts:113-124](../../frontend/src/utils/hierarchy.ts#L113-L124) で定義されています。

| Level | ロール名 | 表示名 | 主な用途 |
|---|---|---|---|
| 1 | `admin` | 管理者 | システム全体の管理者。組織管理・ステージ管理・コンピテンシー管理など全機能にアクセス |
| 1 | `eval_admin` | 評価管理者 | 総合評価（comprehensive evaluation）の専門管理者。総合評価ページのみアクセス |
| 2 | `manager` | 部門マネジャー | 部門単位の管理者。部下の閲覧・承認権限 |
| 3 | `supervisor` | 上司 | 直属上司。部下の目標承認・評価 |
| 4 | `employee` | 従業員 | 一般従業員。自分のデータのみアクセス |
| 5 | `viewer` | 閲覧者 | 読み取り専用。Viewer Visibility による制限付き |
| 6 | `parttime` | パートタイム | パートタイム従業員。最小限のアクセス |

**注意**:
- 同じ Level の `admin` と `eval_admin` は別のロールとして扱われます。一方を持っていてももう一方の権限は持ちません。
- `admin` ロールを持つユーザーは Clerk で自動的に `org:admin` の orgRole が付与されます（[user_service.py:563-577](../../backend/app/services/user_service.py#L563-L577)）。`eval_admin` は DB レベルのみで、Clerk orgRole にはなりません。

## 権限カタログ — 31種類の詳細

各 permission の構造:

- **📂 Group**: マトリクス上の所属グループ
- **📖 説明（DB）**: マトリクスの「説明」列に表示される短い文言
- **🔑 何ができるか**: 実際にこの permission を持つユーザーができる操作
- **🖥️ コード内の参照箇所**: 主な利用ファイル（バックエンド）
- **👥 通常付与されるロール**: 典型的な割り当て先
- **⚠️ 有効化の影響**: ON にしたときに何が変わるか
- **⚠️ 無効化の影響**: OFF にしたときに何が失われるか
- **🔗 関連 permission**: セットで考えるべき他の permission

### User Management（7件）

#### `user:read:all`

- **📂 Group**: ユーザー
- **📖 説明（DB）**: すべてのユーザーを閲覧
- **🔑 何ができるか**: 組織内の全ユーザー一覧を取得・閲覧できる。部下や自分だけでなく、全社員のプロフィールを参照可能。
- **🖥️ コード内の参照箇所**: [user_service.py:136,254,399,535,630,696,751](../../backend/app/services/user_service.py#L136) — `has_permission(Permission.USER_READ_ALL)` で条件分岐
- **👥 通常付与されるロール**: admin
- **⚠️ 有効化の影響**: 対象ロールは組織全体のユーザー一覧（`/org-management` のユーザータブなど）を見られるようになる
- **⚠️ 無効化の影響**: 対象ロールは全体一覧アクセスを失う。部下のみ閲覧（`user:read:subordinates`）または自分のみ（`user:read:self`）にフォールバック
- **🔗 関連 permission**: `user:read:subordinates`（部下のみ）、`user:read:self`（自分のみ）

#### `user:read:subordinates`

- **📂 Group**: ユーザー
- **📖 説明（DB）**: 部下ユーザーを閲覧
- **🔑 何ができるか**: supervisor-subordinate 関係に基づき、自分の直接・間接部下のユーザー情報を閲覧できる
- **🖥️ コード内の参照箇所**: [user_service.py](../../backend/app/services/user_service.py) — マネジャー視点の queries で利用
- **👥 通常付与されるロール**: manager, supervisor
- **⚠️ 有効化の影響**: 部下の閲覧機能（評価フィードバック、ゴール承認など）が動作する
- **⚠️ 無効化の影響**: マネジャー/上司が部下情報にアクセスできなくなり、評価ワークフローが破綻
- **🔗 関連 permission**: `user:read:all`（全体）、`goal:read:subordinates`（部下の目標）

#### `user:read:self`

- **📂 Group**: ユーザー
- **📖 説明（DB）**: 自分のユーザー情報を閲覧
- **🔑 何ができるか**: 自分自身のプロフィール、ステージ、ロールなどを参照できる
- **🖥️ コード内の参照箇所**: 基本的な自己プロフィール取得 APIs
- **👥 通常付与されるロール**: 全ロール（admin, manager, supervisor, employee, viewer, parttime, eval_admin）
- **⚠️ 有効化の影響**: ログインユーザーが自分のプロフィールページを開ける
- **⚠️ 無効化の影響**: 自分の情報すら見られなくなる。**通常無効化してはならない**
- **🔗 関連 permission**: なし

#### `user:manage`

- **📂 Group**: ユーザー
- **📖 説明（DB）**: ユーザーを管理
- **🔑 何ができるか**: ユーザーの作成・全項目編集・削除（実態は status=inactive へのソフトデリート）
- **🖥️ コード内の参照箇所**: [users.py:223,258,282,305](../../backend/app/api/v1/users.py#L223) — `Depends(require_admin)`（admin ロール固定）。サービス層でも [user_service.py:283](../../backend/app/services/user_service.py#L283) で `has_permission(Permission.USER_MANAGE)` がチェックされる
- **👥 通常付与されるロール**: admin のみ
- **⚠️ 有効化の影響**: 対象ロールは組織内のあらゆるユーザーを作成・削除・編集可能。**高リスク権限**
- **⚠️ 無効化の影響**: ユーザー管理機能（org-management のユーザー作成・削除）が使えなくなる
- **🔗 関連 permission**: `user:manage:basic`（基本情報のみ編集可）、`user:manage:plus`（部下構成編集も含む）

#### `user:manage:basic`

- **📂 Group**: ユーザー
- **📖 説明（DB）**: 基本ユーザー情報を編集
- **🔑 何ができるか**: 既存ユーザーの **限定された項目**（name, job_title, department_id）の編集のみ。作成・削除・他項目編集は不可
- **🖥️ コード内の参照箇所**: UserUpdate スキーマに対する permissive update。[user_service.py](../../backend/app/services/user_service.py)
- **👥 通常付与されるロール**: 軽い管理権限が必要なロール（HR スタッフ役など）
- **⚠️ 有効化の影響**: 対象ロールは部下や自分の基本情報を編集可能になる（管理対象範囲は別途決定される）
- **⚠️ 無効化の影響**: 基本情報すら編集できなくなる
- **🔗 関連 permission**: `user:manage`（全項目編集）、`user:manage:plus`（+ 部下構成編集）

#### `user:manage:plus`

- **📂 Group**: ユーザー
- **📖 説明（DB）**: 基本情報と部下構成を編集
- **🔑 何ができるか**: `user:manage:basic` のスーパーセット。基本情報に加えて、`subordinate_ids`（部下構成）も編集可能
- **🖥️ コード内の参照箇所**: [user_service.py](../../backend/app/services/user_service.py) の更新フロー
- **👥 通常付与されるロール**: manager, supervisor（部下構成の管理を委任する場合）
- **⚠️ 有効化の影響**: 対象ロールが組織階層を編集できるようになる
- **⚠️ 無効化の影響**: 部下構成の編集権が失われる
- **🔗 関連 permission**: `user:manage:basic`, `hierarchy:manage`

#### `hierarchy:manage`

- **📂 Group**: ユーザー
- **📖 説明（DB）**: 階層を管理
- **🔑 何ができるか**: supervisor-subordinate 関係（組織階層）の管理操作
- **🖥️ コード内の参照箇所**: `Permission.HIERARCHY_MANAGE` を参照するサービス
- **👥 通常付与されるロール**: admin, manager, supervisor
- **⚠️ 有効化の影響**: 階層の作り変えが可能になる
- **⚠️ 無効化の影響**: 階層変更ができなくなる
- **🔗 関連 permission**: `user:manage:plus`

### Goal（6件）

#### `goal:read:self`

- **📂 Group**: 目標
- **📖 説明（DB）**: 自分の目標を閲覧
- **🔑 何ができるか**: ログインユーザー自身が作成した目標のみ閲覧可能。`/goal-input` の入力済み目標表示、`/past-goal` の自分の過去目標が依存
- **🖥️ コード内の参照箇所**: 目標取得 APIs の権限分岐（`current_user_context.has_permission(Permission.GOAL_READ_SELF)`）
- **👥 通常付与されるロール**: 全ロール（自分の目標参照は基本機能）
- **⚠️ 有効化の影響**: ユーザーが自分の入力済み目標を確認できる
- **⚠️ 無効化の影響**: 自分の目標すら表示されない。`/goal-input` が空表示。**通常無効化しない**
- **🔗 関連 permission**: `goal:manage:self`（編集も含む）、`goal:read:subordinates`（部下分）、`goal:read:all`（全社）

#### `goal:read:all`

- **📂 Group**: 目標
- **📖 説明（DB）**: すべての目標を閲覧
- **🔑 何ができるか**: 組織内すべての目標を横断閲覧。`/admin-goal-list`（全社目標一覧画面）の表示権限
- **🖥️ コード内の参照箇所**: [admin.py:51](../../backend/app/api/v1/admin.py#L51) — `has_permission(Permission.GOAL_READ_ALL)` で明示的にチェックされ、満たさない場合は `PermissionDeniedError`
- **👥 通常付与されるロール**: admin
- **⚠️ 有効化の影響**: 対象ロールが全社目標管理画面にアクセス可能。任意ユーザー・部門・期間で目標を絞り込んで参照可
- **⚠️ 無効化の影響**: `/admin-goal-list` が 403 を返す。全社目標可視化機能が失われる
- **🔗 関連 permission**: `goal:manage`（編集も含む）、`goal:read:subordinates`（部下のみ）

#### `goal:read:subordinates`

- **📂 Group**: 目標
- **📖 説明（DB）**: 部下の目標を閲覧
- **🔑 何ができるか**: 自分の直接・間接部下の目標を閲覧。`/goal-review`（部下目標承認画面）と `/past-goal?subordinateId=...` で利用
- **🖥️ コード内の参照箇所**: [goal_service.py](../../backend/app/services/goal_service.py) — 部下目標一覧 query の分岐で `has_permission(Permission.GOAL_READ_SUBORDINATES)` を使用
- **👥 通常付与されるロール**: manager, supervisor
- **⚠️ 有効化の影響**: 上司・マネジャーが部下の目標を閲覧・承認できる
- **⚠️ 無効化の影響**: 部下目標承認フローが破綻。`/goal-review` が空表示。承認業務不能
- **🔗 関連 permission**: `goal:approve`（承認操作）、`goal:read:all`（全社）

#### `goal:manage`

- **📂 Group**: 目標
- **📖 説明（DB）**: 目標を管理
- **🔑 何ができるか**: 任意のユーザーの目標を作成・編集・削除可能（管理者代行）。本人でなくても目標を変更できる強い権限
- **🖥️ コード内の参照箇所**: [goal_service.py:610,680](../../backend/app/services/goal_service.py#L610) — `is_admin_manage = current_user_context.has_permission(Permission.GOAL_MANAGE)` で本人以外の操作を許可
- **👥 通常付与されるロール**: admin
- **⚠️ 有効化の影響**: 対象ロールが他人の目標を編集・削除できる。**本人の意思に反する変更も可能なため高リスク**
- **⚠️ 無効化の影響**: 管理者代行が不可。本人が忘れた・退職したケースでの修正困難
- **🔗 関連 permission**: `goal:manage:self`（自分のみ）、`goal:read:all`（閲覧）

#### `goal:manage:self`

- **📂 Group**: 目標
- **📖 説明（DB）**: 自分の目標を管理
- **🔑 何ができるか**: 自分自身の目標の作成・編集・削除。`/goal-input` での目標入力・修正がこの権限に依存
- **🖥️ コード内の参照箇所**: [goal_service.py:611,681](../../backend/app/services/goal_service.py#L611) — `existing_goal.user_id` チェックと併用
- **👥 通常付与されるロール**: 全ロール（目標入力は基本機能）
- **⚠️ 有効化の影響**: ユーザーが自分の目標を作成・編集できる
- **⚠️ 無効化の影響**: ユーザーが目標を入力できなくなる。**通常無効化しない**
- **🔗 関連 permission**: `goal:read:self`（閲覧）、`goal:manage`（管理者代行）

#### `goal:approve`

- **📂 Group**: 目標
- **📖 説明（DB）**: 目標を承認
- **🔑 何ができるか**: `/goal-review` 画面での目標承認・差し戻し操作
- **🖥️ コード内の参照箇所**: 目標承認 API endpoint — 承認操作で `has_permission(Permission.GOAL_APPROVE)` を要求
- **👥 通常付与されるロール**: supervisor 以上（admin, manager, supervisor）
- **⚠️ 有効化の影響**: 部下提出の目標を承認・差し戻し可能
- **⚠️ 無効化の影響**: 承認ワークフローが詰まる。部下の目標が承認待ちのまま停滞
- **🔗 関連 permission**: `goal:read:subordinates`（承認対象の閲覧）

### Evaluation（3件）

> ⚠️ **重要な注意**: このグループには「現状コードでチェックされていない権限」が含まれます。
> - `evaluation:read` と `evaluation:review` は **マトリクスで ON/OFF にしても挙動が変わりません**（後述）
> - `evaluation:manage` は **「評価期間（evaluation periods）の管理」を制御**します。評価データそのものではありません

#### `evaluation:read` ⚠️ 未使用

- **📂 Group**: 評価
- **📖 説明（DB）**: 評価を閲覧

## シナリオ集（cookbook）

### シナリオ 1: 新しい「HR スタッフ」ロールを作りたい

人事スタッフが全社のユーザー情報を **閲覧**できるが、編集はできないようにする。

1. `/org-management` → 「ロール」タブで `hr_staff` ロールを作成
2. 「権限」タブで `hr_staff` ロールに以下を ON:

   - `user:read:self`（自分閲覧）
   - `department:read`（部門参照）
   - `stage:read:all`（ステージ参照）
   - `role:read:all`（ロール参照）
3. 編集系（`user:manage*`, `department:manage`, `role:manage` 等）は OFF のまま

- `user:manage:plus` を supervisor ロールに ON にする
- `user:manage`（フル編集）は OFF のまま

- 対象ユーザーに `admin` ロールを **追加**（既存ロールはそのまま、複数ロール所持可）
- 作業完了後にロールを外す
- ⚠️ 注意: `admin` ロール追加時に Clerk の `org:admin` も自動同期される（[user_service.py:563-577](../../backend/app/services/user_service.py#L563-L577)）

### シナリオ 4: eval_admin に総合評価以外の管理機能も付与したい

- 仕様変更が必要な場合は [middleware.ts](../../frontend/src/middleware.ts) の `isAdminRoute` matcher を変更し、対応する admin ページの server guard を更新する必要があります。

## ⚠️ 重要 — 権限マトリクスが制御しないハードコード例外

権限マトリクスが「真の制御源」ではないケースが **2種類**あります:

### A. ハードコード例外（ロール名を直接コードでチェック）

以下の機能は **ロール名を直接コードでチェック**しているため、権限マトリクスの設定を変更しても挙動が変わりません。**変更したい場合は該当ファイルを直接編集する必要があります。**

| 機能 | チェック方法 | ファイル:行 | 修正方法 |

| 総合評価 read（admin-eval-list 閲覧） | `has_any_role(["admin", "eval_admin"])` | [comprehensive_evaluation_service.py:1878-1880](../../backend/app/services/comprehensive_evaluation_service.py#L1878-L1880) | 許可ロールのリストを編集 |
| 総合評価 write（finalize, 設定変更など） | `has_role("eval_admin")` | [comprehensive_evaluation_service.py:1888-1890](../../backend/app/services/comprehensive_evaluation_service.py#L1888-L1890) | ロール名を変更 |
| 評価期間作成・編集 | `@require_role("admin")` | [evaluation_period_service.py:335](../../backend/app/services/evaluation_period_service.py#L335) | デコレータのロール変更 |
| 一部 competency 操作 | `is_admin()` | [competency_service.py:70,84,384,488](../../backend/app/services/competency_service.py#L70) | 追加ロールチェックを実装 |

| 総合評価候補ページの Server Guard | `roles.includes('eval_admin')` | [admin-eval-list/candidates/page.tsx](../../frontend/src/app/(evaluation)/(admin)/admin-eval-list/candidates/page.tsx) | ロール配列を編集 |

### B. 未使用 permission（マトリクスに表示されるが、コードがチェックしていない）

以下の permission は enum と DB migration に定義されていますが、**バックエンドのどこにも `has_permission(...)` 形式の参照が存在しません**。マトリクスで ON/OFF にしても挙動に変化はありません。

| Permission | 状況 | 詳細 |
|---|---|---|
| `evaluation:read` | 🚫 未使用 | enum・bootstrap には存在するが、`has_permission` 参照なし |
| `evaluation:review` | 🚫 未使用 | 同上 |
| `report:access` | 🚫 未使用 | レポート機能自体が未実装の可能性 |
| `stage:read:self` | 🚫 未使用 | 自分のステージは users API で取得されるため不要 |

**確認方法**: `grep -rn 'Permission.<NAME>' backend/app/services backend/app/api` で実用されているか確認できます。

**読者へのメッセージ**: マトリクスで permission を ON/OFF にしても挙動が変わらない場合、（A）ハードコード例外に該当しているか、（B）未使用 permission の可能性が高いです。コードを直接編集するか、未使用 permission の場合は将来実装される可能性を考慮してください。

## FAQ / トラブルシューティング

### Q. マトリクスで権限を変更したのに、ユーザー画面で反映されない

可能性は2つあります:

1. **キャッシュ**: [role_permission_cache.py](../../backend/app/security/role_permission_cache.py) が DB の role_permissions をキャッシュしています。サーバー再起動またはキャッシュ TTL 経過まで反映されない場合があります。
2. **ハードコード例外**: その機能は [第5章のリスト](#️-重要--権限マトリクスが制御しないハードコード例外) に該当している可能性。コードを直接編集する必要があります。

### Q. eval_admin が組織管理画面（org-management）にアクセスできない

**仕様通り**です。eval_admin は総合評価（comprehensive evaluation）専用のロールとして設計されています。組織管理画面は `admin` ロールのみアクセス可能（[middleware.ts](../../frontend/src/middleware.ts) で `org:admin` を要求）。

### Q. カスタムロールを作成したが、想定通り動作しない

カスタムロールは **マトリクス制御の範囲内**でのみ機能します。[第5章のリスト](#️-重要--権限マトリクスが制御しないハードコード例外) の機能はロール名がハードコードされており、カスタムロールには反応しません。

### Q. 「viewer」ロールに特別な挙動がある？

はい。[viewer_visibility.py](../../backend/app/security/viewer_visibility.py) と関連キャッシュにより、viewer ロールには「Viewer Visibility Override」という独立した可視性制御システムが適用されます。これは通常の RBAC とは別の仕組みです。

### Q. admin の Clerk orgRole はどう同期されている？

ユーザーに `admin` ロールが付与/解除されると、[user_service.py:563-577](../../backend/app/services/user_service.py#L563-L577) が Clerk の組織メンバーシップロール（`org:admin` / `org:member`）を自動同期します。eval_admin など他のロールは Clerk と同期されません。

### Q. `user:manage` を付けたのにユーザー削除できない

ユーザー削除は [user_service.py:972](../../backend/app/services/user_service.py#L972) で `require_role("admin")` がハードコードされており、**ロール名「admin」を持つユーザーのみ**実行可能です。`user:manage` permission だけでは不十分。

## 用語集

- **AuthContext**: バックエンドのリクエストごとの認証コンテキスト。ユーザーIDとロール、permissions を保持。[context.py](../../backend/app/security/context.py)
- **RBAC**: Role-Based Access Control（ロールベースアクセス制御）
- **RBAC Helper**: データフィルタリング用のヘルパー。[rbac_helper.py](../../backend/app/security/rbac_helper.py)
- **Permission Catalog**: 全 permission のマスタリスト。DB の `permissions` テーブル
- **role_permissions**: ロール↔permission の紐付けテーブル。マトリクス UI の更新先
- **Viewer Visibility Override**: viewer ロール専用の可視性カスタマイズ機構
- **orgRole**: Clerk の組織メンバーシップロール（`org:admin` / `org:member`）。アプリのロールとは別
- **DB role**: アプリ自身の `roles` テーブルで定義されるロール（admin, eval_admin, manager, ...）
